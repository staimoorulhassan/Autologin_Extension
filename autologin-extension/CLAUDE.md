## High-Level Architecture

This is a **Firefox MV3 WebExtension** (also Chrome-compatible) for automated cross-site login. It compiles to three entry points via webpack:

| Entry | Source | Role |
| --- | --- | --- |
| `background.js` | `src/background/worker.ts` | Service worker — message router, DB owner, batch login orchestrator, login state tracking |
| `popup.js` | `src/popup/popup.tsx` | React 18 UI shown from the toolbar — credential CRUD, batch controls, log viewer |
| `content.js` | `src/content/contentMain.ts` | Injected at `document_start` on all URLs — form detection, fill, submit, screenshot, CAPTCHA detection |

The three components communicate exclusively via `chrome.runtime.sendMessage`. All shared state lives in IndexedDB (owned by the background worker) or `chrome.storage.session` (for transient batch state).

### Module Map

```
src/
├── background/      worker.ts — chrome.runtime.onMessage dispatcher; the only owner of DB writes
├── content/         contentMain.ts — DOM-side handlers; never touches DB directly
├── popup/           popup.tsx + components/, forms/, hooks/, context/ — React UI
├── messaging/       Typed message system (see "Messaging" below)
├── store/           database.ts — Dexie schema + per-table CRUD stores
├── crypto/          encryption.ts — TweetNaCl secretbox (XSalsa20-Poly1305 authenticated encryption)
├── automation/      engine.ts (form detection/fill), captcha.ts, anti-detection.ts, multi-account.ts, proxy.ts
├── monitoring/      logger.ts, monitor.ts, alerts.ts
└── types/           index.ts — Credential, Cookie, LoginLog, Screenshot, LoginStatus, FormFields, ...
```

### Messaging — important

There are **two messaging modules** in `src/messaging/`. Use the typed one:

- **Use `messenger.ts` + `handlers.ts` + `types.ts`** (re-exported via `messaging/index.ts`). Imports from `@messaging/index`. This is the discriminated-union, Promise-wrapped, timeout-aware system actually wired into `worker.ts`, `contentMain.ts`, and `popup.tsx`. Background message types are listed in `BackgroundMessageType` (`GET_CREDENTIALS`, `START_BATCH_LOGIN`, etc.); content message types in `ContentMessageType` (`DETECT_FORM`, `FILL_FORM`, etc.).
- **Avoid `messageSystem.ts`** — it's an earlier Task 3 design kept for its tests, not used by the live entry points.

Key entry points: `sendToBackground(msg)` from popup, `sendToContent(tabId, msg)` from background, `registerHandler(MESSAGE_TYPES.X, fn)` + `dispatchMessage` for handlers. Default timeout is 5s — login flows that wait on form submission use longer timeouts (see `worker.ts` for the actual values).

### Data Storage

`src/store/database.ts` defines `AutoLoginDB` (Dexie) with four tables:

- `credentials` — `++id, url, created_at` (passwords stored encrypted in `password_encrypted`)
- `cookies` — composite key `[account_id+name]`, indexed by `account_id`, `expires`
- `logs` — `++id, account_id, timestamp, status`
- `screenshots` — `++id, account_id, timestamp, stage`

`credentialStore.delete(id)` performs a **cascade delete** across cookies/logs/screenshots — preserve this when adding new related tables.

### Crypto

`src/crypto/encryption.ts` uses `tweetnacl` `secretbox` (despite the `_ALGORITHM = 'AES-256-GCM'` comment, the implementation is XSalsa20-Poly1305 — both are authenticated encryption with 256-bit keys). Format: 24-byte nonce + ciphertext, hex-encoded. `deriveKeyFromPassword` uses iterated NaCl hashing (1000 rounds) — there's a `TODO: PBKDF2/Argon2` comment for production hardening.

## TypeScript & Path Aliases

Strict mode is on (`noImplicitAny`, `strictNullChecks`, `noUnusedLocals`, `noUnusedParameters`, etc.). Aliases configured in both `tsconfig.json` and `webpack.config.js`:

```
@/*           → src/*
@automation/* → src/automation/*
@content/*    → src/content/*
@popup/*      → src/popup/*
@store/*      → src/store/*
@crypto/*     → src/crypto/*
@types/*      → src/types/*
@messaging/*  → src/messaging/*
```

**`@managers/*` is also aliased but `src/managers/` does not exist** — leftover from the original spec. Don't import from it; use `@store` / `@messaging` instead. The README's mention of a `managers/` directory is stale.

`ts-loader` is configured with `transpileOnly: true`, so webpack does not type-check. Run `npm run type-check` separately, or rely on CI.

## Testing Conventions

- Jest with `ts-jest` preset, `jsdom` environment, root at `src/`.
- Setup in `src/__tests__/setup.ts` mocks `chrome.runtime`, `chrome.storage.local/sync`, and `chrome.tabs` globally — extend that file rather than re-mocking per test.
- Tests live in `src/__tests__/<module>/` mirroring `src/<module>/` (`crypto/`, `store/`, `messaging/`, `automation/`, `popup/`, `monitoring/`).
- Coverage threshold of 85% is enforced globally; `popup.tsx`, `worker.ts`, and `contentMain.ts` are excluded from coverage as thin entry points.
- Test timeout is 30s.

Existing patterns (180+ tests across crypto/store/messaging) follow Jest standard practice — there are project-specific patterns documented in `~/.claude/projects/.../memory/encryption-db-testing-patterns.md` if you have memory access.

## Manifest & Permissions

`manifest.json` is **MV3** with:

- `permissions`: `storage`, `tabs`, `scripting`, `offscreen_document`, `cookies`, `downloads`, `alarms`
- `host_permissions`: `<all_urls>` (required for content script injection on any login page)
- `optional_permissions`: `proxy`
- Background as ES module (`"type": "module"`); content script runs at `document_start`, no frames.

Webpack copies `manifest.json` and `public/` (popup.html, options.html, options.js) into `dist/` on each build.

## Methodology Notes

The project follows **Specification-Driven Development (SDD)** — every feature traces back to a numbered Task in `TASK-N-*.md` files at the repo root, with acceptance criteria and tests written first. `PHASE-2-STATUS.md` is the current authoritative progress tracker. When implementing a new task, find its `TASK-N-COMPLETE.md` (if it exists) for the agreed scope and acceptance criteria; otherwise consult `PHASE-2-STATUS.md`.
